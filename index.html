对设计模式的一些思考

Posted on 2017-03-19

最近在看<>一书来学习设计模式,不得不说这本书相比于比较枯燥难懂的 GOF 出产的设计模式容易理解得多,书中使用了很多有趣的生活例子和插图来解释一个个看似难懂的设计模式,让我对设计模式有了更深一层的理解,因此也推荐大家去阅读一下.

在花了一个月去阅读完这本书之后,我对设计模式充满了热情,于是逐渐开始用于工作和自己的项目中去.比如我们公司的 app 的下单逻辑,下单有多种渠道(网页/app/微信商城),这个下单的过程是相同的,但是下单后的逻辑会略有不同,这个时候使用工厂模式去重构这段代码就最好不过了,经过一轮重构,代码变得更加优雅和容易维护,但缺点就是多了几个子类,但是这个代价是值得的.

但是,设计模式也给我带来了一点麻烦,由于我对设计模式有点走火入魔,甚至一些很小的功能我也想去使用设计模式重构,但是由于缺乏单元测试的支持,导致线上一个老问题又重现,所幸的这并非是一个重要业务.

由此我得出一些思考和感想,设计模式的”模式”其实并不重要,也就是说设计模式并不一定要是工厂模式/策略模式/单例模式等等,其实更应该领会的是它的原则和理念.

1.符合”少用继承,多用组合”的原则.由于领取红包和返现其实都是下单的额外行为,如果放在下单父类,那么所有子类都要继承或实现它,但并非所有子类都需要的,因此我把这两种行为拆分成独立的类,并且组成一个继承体系,每个行为子类都包含一个逻辑方法,我们的下单子类只要通过setCashbackBehavior和setGetRedPacketBehavior方法来设置特定的返现和领取红包行为.假如不需要,就不去调用这个方法.

2.符合”针对接口编程”的原则.在重构后的代码,我把 Client 类的createOrder方法的参数改成了OrderCreator $creator这样做的好处是:第一,能够解决类型安全问题.第二,去掉了臃肿的 switch 分支,这样以后无论新增多少下单子类,我都不需要修改Client 的代码.

3.符合”对扩展开放,对修改关闭”原则.像上面一点说了,createOrder使用OrderCreator $creator作为参数,即使后面新增多少下单子类都是不需要修Client 代码的,同时行为的剥离能够让下单子类拥有运行时改变行为的能力,比如我们 AppOrderCreator的返现逻辑需要改成 web 下单的返现逻辑,只需要修改一行代码$app->setGetRedPacketBehavior(new WebCashbackBehavior($app));就能在运行时修改行为了,其中不需要动到AppOrderCreator的一行代码.

尽管优点很多,但是缺点也是有的,比如我们维护的类变得更多更细了,但是这在于一个大项目来说并不算什么,但是放在一个小型并且不会经常扩展的项目,这种做法其实没必要的.

设计模式更多是提供思想,而非具体的做法和规范,很多时候千万不要为了设计模式而设计模式,这很容易会把原来就没什么问题的代码搞得更加乱七八糟和难以维护,设计模式更应该作为一种思想来指导我们写出低耦合和高内聚的代码.上面代码和逻辑可能还存在问题,希望各位能够给出更多的指点和修正,同时也极力推荐各位阅读<>~

  3. 把不用的大变量uset掉,节约内存.

使用Redis的bitmap来优雅地实现统计活跃用户数

Posted on 2016-10-19

在日常的开发中,活跃统计是比较常见的功能,比较常用的做法是会做一个数据表,然后存储目前登录的用户.功能很简单,但是使用数据库对于并发高的应用来说是代价挺大的,能不能换一种思路去实现这些功能呢?

bitmap算法

所谓bitmap就是用一个bit位来标记某个元素对应的value，而key即是这个元素。由于采用bit为单位来存储数据，因此在可以大大的节省存储空间.

Redis的bitmap

自己去实现bitmap算法是件麻烦的事情,幸好,伟大的redis再次出现,它内置了一套bitmap的操作,例如setbit/getbit/bitcount这几种方法,下面简单介绍一下使用这些方法来实现签到和活跃用户的功能.

首先,我们来实现活跃用户统计,那么我们可以设置bitmap的key为active_user来表示.这个bitmap不需要预先创建,假如用户id为111的用户登录了,只需要调用方法setbit active_user 111 1 这个意思是把bitmap对应的位置(111)设置为1,那么其他用户登录同样也是这个道理,最后就会形成下图这样:

那么,我们怎么去统计这个bitmap呢?bitcount能帮到你,只需要调用bitcount active_user得到的结果就是目前活跃的用户数了,同样道理,只要退出就把对应user_id位置的bitmap设置为0,如用户111退出了setbit active_user 111 0 ,就是如此简单.

优势在哪

其实这么简单的一个功能,很多方案也可以解决,甚至用redis的set方法也能实现,那为什么这个方法值得使用呢?因为上面提过bitmap算法是使用bit来存储数据的,除了能够节省很大一笔空间之外,还能拥有极高的性能,它的时间复杂度是O(1),操作改变的速度极快,getspool.com这个网站正是使用这种方法来统计,网站称,在模拟1亿2千8百万用户的模拟环境下,在一台MacBookPro上.典型的统计如“日用户数”(dailyunique users) 的时间消耗小于50ms, 占用16MB内存。想想大部分网站也比较难达到1亿2千八百万的用户数量,所以这个方法值得实施.

使用swoole,mongodb异步记录日志

Posted on 2016-10-03

起因

自己的创业项目中,原来的日志写入是用了yii2的默认写入方式,也就是写入文件的方式,这种方式简单粗暴,也是最原始的写入方式,但是有时候要去定位日志错误,真的是一件很烦恼的事情,特别是现在并发访问量高的时候,日志刷新速度很快,而且占用磁盘也多,服务器的io同时也增高了,直接让服务器的压力增大.

拆出来

因此,决定把日志独立出来,也就是存储方式改为db存储,一开始考虑了mysql,但目前mysql只有一台服务器,如果用mysql做日志存储系统,那么高并发的写入也一样会影响到平时的业务写入,因为写入频繁,会导致mysql的行所增多,所以mysql就否认了.mongodb是一个不错的选择,由于是结构化存储,很适合日志这种无规则,大规模的写入.

异步写入

尽管mongodb是独立一台服务器,但是频繁同步的日志写入,也一样要求服务器不断去连接mongodb,插入mongodb,这也会导致一部分的网络开销,因此我也想到了异步的方式,
说到异步的第三方组件,可是很多很多的,比较常见的就是使用消息队列(RabbitMq,ActiveMq),但是要为了一个日志系统,引入一个比较重的消息队列,就不太值得了,于是想到
了现在很火的swoole.swoole的学习成本还是挺高的,如果对网络编程这块不熟,很多东西都是模模糊糊,比如什么是进程和线程的区别,什么是worker/reactor,这些概念
还是需要一定基础支撑的,但是没关系,本次我的需求很简单,我只需要用到swoole的task来实现异步写入日志就可以了.

swoole的task介绍

swoole的业务逻辑部分是同步阻塞运行的,如果遇到一些耗时较大的操作,例如访问数据库、广播消息等,就会影响服务器的响应速度.因此Swoole提供了Task功能,
将这些耗时操作放到另外的进程去处理,当前进程继续执行后面的逻辑.这是官方对task功能的一个简述,那么正好适合于我们去做异步写入日志的功能.

商城秒杀系统构思系列（一）--超卖的解决方案

Posted on 2016-09-27

今天中午，组长找我聊天，说我们的商城可能准备做一个秒杀系统，按照他的说法，目前的系统架构，支撑个百来并发量都是不太行的，于是先和我商量关于这个秒杀系统的想法和一些做法。听到这个消息，说实话我是挺激动的，毕竟工作一年多了，这方面的概念理论就看多了，但是还是缺乏一定的实践经验，于是下定决心去做好它，于是就开始我的第一步构思：秒杀—如何防止超卖？

数据库事务

超卖是秒杀一个首要解决的问题，现在的商城就是会出现超卖情况的，很多用户下单成功了，但是商品库存居然变为负数了，这就涉及到数据库一个事务问题，看过以前的代码，下单是根本没有做任何防止并发请求的处理的，我的第一个想法就是数据库事务，但是使用事务虽然是最简单的，但是也是最耗损性能的，或许现在这个秒杀只是为了撑个百来个并发，但是做程序也应该想到扩展性和未来的业务发展，所以我就否定了数据库事务这个想法。

Redis

redis一直只是作为我的缓存工作使用，但是redis的强大之处远非缓存，因此查阅了一些资料，得知redis是单线程的，而且他的操作都是原子性的，换做人话来说就是能够保持数据一致。为了性能和库存的准备实时性，库存的控制应该放在缓存层来做而非数据库层，因为无论读还是写，redis都会比mysql强大许多。

具体实施

这里我将会用到redis的一个原子性方法decr。 这个方法是可以递减某个key对应的value的，并且是保持一致的，也就是无论多少请求到达，都会只去顺序减去一次，有类似队列的功效，下面说说我的操作步骤。

  1. 把所有要秒杀的商品的库存事先写到缓存中去，如商品ID为1的key就是goods1这样
  2. 代码中，判断是否抢空，就使用

$result = $redis->decr('goods1')

这句代码会返回goods被减去后的数量，也就是当商品可能剩下0的时候，那么这$result变量就会是-1了，那么我们就可以告知用户，这个商品已经被抢光了。
3.后续的请求都是判断，如果这个$result少于0都是表示库存完结了。

不完美的地方

这个判断在ab用并发请求测试过，基本上都不会出现负数库存的情况，但是这个只是针对每次请求只能购买一件的情况来设计的，如果是多件要这样设计就走不通了，得根据产品最终的需求和协商再来构思这块如何能够做的更好。

yii使用自带组件pjax的一个小坑

Posted on 2016-09-22
  • 其实是很久之前遇到的问题了，但发现忘记写个总结了，今天看yii2群里的一个同学也出现这个问题，因此又想起来了。
  • 事情是这样的，由于家里网速很慢，因此使用pjax的时候经常会请求超过1秒以上，而恰好yii自带的pjax组件又是设置超时时间是一秒的！因此每次pjax请求超时都会自动转为普通请求，当时调了半天也不知什么原因，后来才了解到pjax的超时机制，最后把组件的timeout调高之后就一切ok了。
